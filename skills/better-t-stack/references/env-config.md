# BTS Environment Config: @t3-oss/env

## Pattern

BTS projects use `@t3-oss/env-nextjs` (for Next.js) and `@t3-oss/env-core` (for native/CLI) to validate environment variables at startup with Zod schemas.

The `packages/env` package is split by runtime to enforce explicit boundaries — server secrets must never reach client or native bundles.

---

## `packages/env/src/server.ts`

Server-only variables. Never imported by client code or native app.

```typescript
import { createEnv } from "@t3-oss/env-nextjs"
import { z } from "zod"

export const serverEnv = createEnv({
  server: {
    NODE_ENV: z.enum(["development", "test", "production"]),
    DATABASE_URL: z.string().url(),
    BETTER_AUTH_SECRET: z.string().min(32),
    BETTER_AUTH_URL: z.string().url(),
    RESEND_API_KEY: z.string().startsWith("re_"),
    STRIPE_SECRET_KEY: z.string().startsWith("sk_"),
    STRIPE_WEBHOOK_SECRET: z.string().startsWith("whsec_"),
  },
  // Uses process.env — requires Next.js or Node.js runtime
  runtimeEnv: process.env,
  emptyStringAsUndefined: true,
})
```

## `packages/env/src/web.ts`

Next.js client-side variables. Only `NEXT_PUBLIC_*` can be accessed from client components.

```typescript
import { createEnv } from "@t3-oss/env-nextjs"
import { z } from "zod"
import { serverEnv } from "./server"

export const webEnv = createEnv({
  extends: [serverEnv],
  client: {
    NEXT_PUBLIC_APP_URL: z.string().url(),
    NEXT_PUBLIC_BETTER_AUTH_URL: z.string().url(),
    NEXT_PUBLIC_STRIPE_PUBLISHABLE_KEY: z.string().startsWith("pk_"),
  },
  runtimeEnv: {
    NEXT_PUBLIC_APP_URL: process.env.NEXT_PUBLIC_APP_URL,
    NEXT_PUBLIC_BETTER_AUTH_URL: process.env.NEXT_PUBLIC_BETTER_AUTH_URL,
    NEXT_PUBLIC_STRIPE_PUBLISHABLE_KEY: process.env.NEXT_PUBLIC_STRIPE_PUBLISHABLE_KEY,
  },
  emptyStringAsUndefined: true,
})
```

## `packages/env/src/native.ts`

Expo native variables. Only `EXPO_PUBLIC_*` variables are safe here.

```typescript
import { createEnv } from "@t3-oss/env-core"
import { z } from "zod"

export const nativeEnv = createEnv({
  clientPrefix: "EXPO_PUBLIC_",
  client: {
    EXPO_PUBLIC_API_URL: z.string().url(),
    EXPO_PUBLIC_BETTER_AUTH_URL: z.string().url(),
  },
  runtimeEnv: {
    EXPO_PUBLIC_API_URL: process.env.EXPO_PUBLIC_API_URL,
    EXPO_PUBLIC_BETTER_AUTH_URL: process.env.EXPO_PUBLIC_BETTER_AUTH_URL,
  },
  emptyStringAsUndefined: true,
})
```

---

## Import Rules

| From | Import |
|---|---|
| `packages/api` (server) | `@myapp/env/server` or `@myapp/env` |
| `apps/web` server component / Route Handler | `@myapp/env/server` |
| `apps/web` client component | `@myapp/env/web` (NEXT_PUBLIC_ only) |
| `apps/native` | `@myapp/env/native` (EXPO_PUBLIC_ only) |

**Never import `@myapp/env/server` from a client component or native app.**

## `bts.jsonc`

The `bts.jsonc` file at the repo root is generated by the BTS CLI and tracks which options were selected at scaffold time. It is read by `bunx create-better-t-stack add` to know the existing stack.

```jsonc
{
  // Created: 2026-03-06T10:00:00.000Z
  // Version: 3.22.1
  "frontend": ["next", "native-uniwind"],
  "backend": "self",
  "database": "sqlite",
  "orm": "prisma",
  "auth": true,
  "addons": ["biome", "lefthook", "mcp", "ruler", "skills", "turborepo"],
  "examples": true
}
```

Do not edit `bts.jsonc` manually. It is safe to delete if the `add` command is no longer needed.
