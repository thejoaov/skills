# BTS Stack Choices: ORM, Database, Package Manager

## ORM: Prisma vs Drizzle

### Prisma

**Use when:**
- The project runs on a long-lived server (Next.js on Vercel, self-hosted Node)
- You want a schema-first workflow with a single source of truth (`schema.prisma`)
- The team benefits from Prisma Studio for DB inspection
- You need relationship-heavy queries with automatic join resolution

**Setup in BTS:**
- Schema at `packages/db/prisma/schema/schema.prisma` (multi-file schema recommended)
- Client generated to `packages/db/generated/` or the default location
- db commands: `db:generate`, `db:push`, `db:migrate`, `db:studio`

**Key patterns:**
```typescript
// packages/db/src/client.ts
import { PrismaClient } from "@prisma/client"

const globalForPrisma = globalThis as unknown as { prisma: PrismaClient }

export const prisma =
  globalForPrisma.prisma ??
  new PrismaClient({
    log: process.env.NODE_ENV === "development" ? ["query"] : [],
  })

if (process.env.NODE_ENV !== "production") globalForPrisma.prisma = prisma
```

**Adapters for edge/serverless:**
- `@prisma/adapter-pg` — for Postgres via `pg` driver (Supabase, Neon)
- `@prisma/adapter-libsql` — for Turso/libSQL
- `@prisma/adapter-d1` — for Cloudflare D1

### Drizzle

**Use when:**
- The project runs on edge runtimes (Cloudflare Workers, Turso, Vercel Edge)
- You want SQL-like query building with full TypeScript inference
- You need fine-grained query control without an ORM abstraction layer
- Migrations must be plain SQL files (not generated by an ORM CLI)

**Setup in BTS:**
- Schema at `packages/db/src/schema/`
- Client configured per driver (libSQL, pg, etc.)
- Migrations in `packages/db/drizzle/`

**Key patterns:**
```typescript
// packages/db/src/client.ts (Turso/libSQL)
import { createClient } from "@libsql/client"
import { drizzle } from "drizzle-orm/libsql"
import * as schema from "./schema"

const client = createClient({
  url: process.env.DATABASE_URL!,
  authToken: process.env.DATABASE_AUTH_TOKEN,
})

export const db = drizzle(client, { schema })
```

## Database: SQLite/Turso vs Postgres

### SQLite + Turso (edge/small projects)

| Aspect | Notes |
|---|---|
| Local dev | SQLite file — zero setup |
| Production | Turso (libSQL) — edge replicas |
| ORM | Drizzle (preferred) or Prisma with adapter |
| Best for | Small apps, solo projects, edge-first workloads |
| Auth compatibility | Better Auth supports both |

```bash
# .env for local dev
DATABASE_URL="file:./dev.db"

# .env for production
DATABASE_URL="libsql://your-db.turso.io"
DATABASE_AUTH_TOKEN="..."
```

### PostgreSQL + Supabase / Neon

| Aspect | Notes |
|---|---|
| Local dev | Needs a running Postgres instance (Docker or Supabase local) |
| Production | Supabase (with storage + auth) or Neon (serverless Postgres) |
| ORM | Prisma (preferred) or Drizzle with `drizzle-orm/node-postgres` |
| Best for | Production apps, complex queries, full relational model |

```bash
# .env
DATABASE_URL="postgresql://user:pass@localhost:5432/myapp"
DIRECT_URL="postgresql://user:pass@localhost:5432/myapp"  # for Prisma migrations via Supabase
```

## Package Manager: Bun vs pnpm

### Bun (personal default)

- Unified runtime + package manager — no need for `tsx`, `ts-node`, or separate Node.js
- `bun.lockb` (binary) lockfile — faster and deterministic
- Native test runner (`bun test`) — no extra dependency for simple tests
- Workspace catalog via `catalog:` in root `package.json`
- Script execution: `bun run <script>` or `bun <file>.ts` directly

**Caveats:**
- Some native Node.js addons may not work with Bun's runtime
- Bun's test runner is less feature-complete than Vitest for complex setups

### pnpm (alternative)

- Strict `node_modules` hoisting rules — prevents phantom dependencies
- `pnpm-workspace.yaml` for workspace config
- Catalog via `catalog:` in `pnpm-workspace.yaml` (same syntax as Bun catalog)
- Better compatibility with legacy Node.js ecosystem packages

**When to use pnpm:** Projects with heavy native addon dependencies, or when deploying to environments where Bun runtime isn't supported (e.g., some Lambda runtimes).
